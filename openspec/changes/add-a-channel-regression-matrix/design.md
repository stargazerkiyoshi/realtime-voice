## 背景

A 通道已具备 WebSocket 会话、流式 ASR/LLM/TTS、打断等关键行为，但缺少标准化回归体系。  
目前痛点主要是：

- 场景集合分散，覆盖范围不稳定。
- 运行入口不统一，本地与 CI 行为不一致。
- 结果不可结构化比对，历史趋势难追踪。
- 通过/失败标准未固化，结论依赖执行者经验。

## 目标 / 非目标

- 目标：
  - 固化 10 类核心生存场景，形成稳定回归矩阵。
  - 提供单一回归入口，支持全量执行与按场景执行。
  - 输出统一结构化结果（JSON），供本地与 CI 消费。
  - 为每个场景定义明确判定条件（必需事件、禁止事件、时序约束）。
- 非目标：
  - 不覆盖 B/C 通道（SIP/电话）回归。
  - 不在本变更中重构会话内核或 WebSocket 协议。
  - 不引入重型测试编排框架。

## 决策

- 决策 1：采用“场景注册表 + 通用执行器”模型。
  - 每个场景声明：`id`、`name`、`setup`、`steps`、`assertions`、`timeoutMs`。
  - 通用执行器负责生命周期、日志采集、判定收敛。
- 决策 2：统一入口使用单命令触发（例如脚本命令），并支持参数化执行。
  - 全量：执行 10 类场景。
  - 定向：按 `scenario id` 执行子集。
- 决策 3：结构化输出采用单文件 JSON（必要时可扩展 JSONL）。
  - 顶层包含运行元信息（时间、分支、提交、总耗时、汇总状态）。
  - 场景维度包含断言详情、失败原因、关键事件时间线。
- 决策 4：判定条件以“必需条件 + 禁止条件 + 超时”三元模型表达。
  - 必需条件缺失即失败。
  - 禁止条件出现即失败。
  - 超时未达成预期即失败。
- 决策 5：CI 初期采用非阻塞模式运行回归（输出结果与告警），待稳定后再升级为阻塞。
- 决策 6：性能阈值纳入回归结果，但采用分阶段门禁策略。
  - 阶段 A（当前）：仅采集并输出阈值评估结果，不作为失败条件。
  - 阶段 B（稳定观察期后）：在 nightly 中作为软门禁（告警/标记不通过，不拦截主流程）。
  - 阶段 C（稳定后）：升级为 PR 级硬门禁（阈值不达标即失败）。
  - 初始关注指标：`speech_end -> first_audio`（p50 / p95）。
  - 阈值策略：采用统一阈值，不区分本地与 CI 环境。
  - 初始阈值：`p50 <= 900ms`，`p95 <= 1500ms`。

## 10 类场景与判定要点

1. 连接与会话生命周期：必须出现 `ready` 和 `end`，顺序正确。
2. 未 start 先 audio：必须返回 `error(NO_SESSION)`。
3. 音频到 ASR：必须出现至少一次 `asr` 事件。
4. ASR final 到 assistant：必须出现 `asr.is_final=true` 后的 `assistant`。
5. assistant 到 tts：必须在 assistant 后出现 `tts`。
6. 打断：SPEAKING 中注入语音后必须出现 `barge_in`，且旧回复停止增长。
7. close_mic 恢复：`close_mic` 后可继续新一轮输入并产生可观测输出。
8. ping/pong：发送 `ping` 必须在阈值内收到 `pong`。
9. 非法消息：发送非法 payload 必须返回标准错误且进程保持可服务。
10. 断连清理：客户端主动关闭后，会话应在可接受时间内收口。

## 风险 / 取舍

- 风险：外部 ASR/LLM/TTS 服务波动导致场景偶发失败。
  - 缓解：场景区分“协议级必需断言”和“内容级弱断言”，并记录失败分类（环境/逻辑）。
- 风险：实时系统时序抖动导致误判。
  - 缓解：统一超时与重试策略，保留关键事件时间线用于复判。
- 取舍：先做最小可用矩阵（10 类核心），不一次性覆盖全部边角场景。

## 迁移计划

1. 建立场景注册表与 10 类场景定义。
2. 实现统一入口与执行器（全量/子集）。
3. 增加结构化结果输出与汇总状态。
4. 将回归入口接入 CI（先非阻塞，再按稳定性升级为阻塞）。
5. 接入性能阈值评估并按 A/B/C 三阶段逐步升级门禁。
