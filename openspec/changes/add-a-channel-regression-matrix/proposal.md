# 变更：建立 A 通道稳定性回归场景矩阵

## 为什么

当前 A 通道（WebSocket）已有核心链路能力，但回归验证仍以手工检查和零散脚本为主，缺少统一入口、统一结果格式、统一判定标准。  
这会导致稳定性问题难以及时发现，且不同人执行回归时结论不可比对。

## 变更内容

- 定义并固化 A 通道 10 类核心生存场景（覆盖连接、会话、音频、打断、异常路径）。
- 提供统一回归运行入口，支持一次性执行全部场景与按场景筛选执行。
- 输出结构化回归结果（机器可读），用于本地排查与 CI 集成。
- 明确每个场景的通过/失败判定条件，避免“靠人工理解”的模糊结论。

## 10 类核心生存场景（已确认基线）

1. 连接与会话生命周期：`connect -> start -> ready -> stop -> end`
2. 非法时序：未 `start` 先发 `audio` 必须返回 `NO_SESSION`
3. 音频上行与识别链路：`audio` 可触发 ASR 输出（partial/final）
4. 最终识别到回复：ASR final 后可触发 assistant 流式输出
5. 回复到合成播放：assistant 输出后可触发 TTS 音频下发
6. 打断语义：SPEAKING 期间开口触发 `barge_in` 并中断当前输出
7. `close_mic` 恢复能力：关闭麦克风后可继续后续轮次
8. 保活链路：`ping` 必须返回 `pong`
9. 非法消息健壮性：无效消息触发标准错误返回，不导致进程崩溃
10. 断连清理：客户端关闭连接后会话能正确收口

## 影响范围

- 受影响规格：`a-channel-regression`（新增）
- 受影响代码：回归测试入口、场景定义、结果输出模块、CI 流程（仅接入入口，不引入复杂编排）
- CI 策略：初期采用非阻塞执行（失败告警但不拦截合并）
- 非目标：不在本变更中引入 B/C 通道场景，不改动现有 WebSocket 协议语义
