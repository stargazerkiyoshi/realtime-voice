## 新增需求

### 需求：A 通道核心生存场景矩阵
系统必须定义并维护 A 通道稳定性回归的 10 类核心生存场景，且每次回归执行必须覆盖完整场景集合或明确声明筛选子集。

#### 场景：全量矩阵定义
- **当** 维护 A 通道回归规范
- **则** 系统必须包含以下 10 类场景定义：连接会话生命周期、未 start 先 audio、音频到 ASR、ASR final 到 assistant、assistant 到 tts、打断、close_mic 恢复、ping/pong、非法消息健壮性、断连清理

### 需求：统一回归运行入口
系统必须提供统一回归运行入口，支持一次性执行全部场景以及按场景标识执行子集，并输出统一运行摘要状态。

#### 场景：统一入口执行
- **当** 执行 A 通道回归
- **则** 系统必须通过同一入口触发执行，并支持全量模式与按场景筛选模式，最终给出整体通过/失败结论

### 需求：结构化结果输出
系统必须输出机器可读的结构化回归结果，至少包含运行级元信息、场景级状态与断言级明细。

#### 场景：结果可消费
- **当** 回归执行结束
- **则** 系统必须输出结构化结果，包含运行时间、提交标识、每个场景状态、失败原因与关键事件时间线

### 需求：场景判定条件显式化
系统必须为每个场景定义显式通过/失败判定条件，并使用一致的判定模型执行（必需条件、禁止条件、超时条件）。

#### 场景：判定一致性
- **当** 任一场景执行
- **则** 系统必须依据该场景预定义判定条件自动得出 pass/fail，且判定依据可追溯
